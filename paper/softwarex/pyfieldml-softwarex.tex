\title{pyfieldml: a modern Python implementation of the FieldML 0.5 standard for computational biomechanics}

\begin{abstract}
FieldML is a declarative standard for representing mathematical fields over finite-element meshes, developed under the Physiome Project. Its reference C++ implementation has been largely unmaintained since 2015, leaving modern Python biomechanics pipelines without a native FieldML path. \texttt{pyfieldml} is a pure-Python reimplementation of FieldML 0.5 that exposes a full evaluator-graph DOM, an evaluation engine covering Lagrange and cubic-Hermite bases on the six standard topologies, a ten-dataset musculoskeletal model zoo, interoperability bridges to meshio, PyVista, XDMF and scikit-fem, a command-line interface with validation linter, and an in-browser JupyterLite demo. The package is \texttt{pip}-installable, Apache-2.0 licensed, and has 220+ continuously-integrated tests on Python 3.10--3.13 across Linux, macOS and Windows.
\end{abstract}

\section{Motivation and significance}

Computational physiology and biomechanics depend on faithful, reusable descriptions of geometric and physical fields defined over finite-element meshes: bone and soft-tissue geometry, muscle-fibre orientations, bone-mineral density, cardiac conductivity tensors. The Physiome Project \cite{hunter_borg_2003}, coordinated from the Auckland Bioengineering Institute, proposed FieldML as the standard representation for such data. The conceptual paper by Christie et al.~\cite{fieldml_christie_2009} introduces FieldML as ``a declarative language for building hierarchical models from reusable, self-contained components,'' in which a document is an explicit graph of typed \emph{evaluators} (parameter, external, aggregate, piecewise, reference) over discrete \emph{ensemble} and continuous domains. This evaluator-graph view is powerful: it captures not just a mesh and a field but the entire mathematical pipeline that produces field values from parametric coordinates, including basis functions, scale factors, and connectivity.

The reference C++ implementation of FieldML, FieldML-API \cite{fieldml_api}, was developed alongside the 0.5 specification and is tightly coupled to the OpenCMISS solver infrastructure \cite{bradley_opencmiss_2011}. It depends on libxml2 and HDF5, must be built from source, exposes no Python bindings, and has seen little maintenance since approximately 2015. Meanwhile, the computational-physiology and biomechanics communities have moved decisively toward Python: meshio \cite{meshio} for format interoperability, PyVista \cite{pyvista} for visualisation, scikit-fem \cite{scikit_fem} for lightweight FEM assembly, and the OpenSim \cite{delp_opensim_2007,seth_opensim_2018} family of tools for musculoskeletal (MSK) dynamics are now standard building blocks. No native Python path into FieldML currently exists. As a consequence, the substantial corpus of Physiome-Project models encoded in FieldML 0.3, 0.4 and 0.5 is effectively invisible to modern Python pipelines, and new investigators must choose between ad-hoc formats or a steep C++ build.

\texttt{pyfieldml} closes this gap. It is a pure-Python, independent reimplementation of FieldML 0.5 that can be installed with \texttt{pip install pyfieldml} in a single step, with no compilation required. It reads and writes FieldML 0.3, 0.4 and 0.5 (up-converting older versions transparently), evaluates fields at arbitrary parametric coordinates using a plugin-based basis-function registry, and round-trips documents through meshio, PyVista, XDMF3 and scikit-fem. The concepts of the 2009 paper \cite{fieldml_christie_2009} map directly onto the package's Python class hierarchy: \texttt{ParameterEvaluator}, \texttt{ExternalEvaluator}, \texttt{AggregateEvaluator}, \texttt{PiecewiseEvaluator} and \texttt{ReferenceEvaluator} are first-class objects composed into a \texttt{Document}. Related open-source mesh infrastructure--- libMesh \cite{kirk_libmesh_2006} and the FEniCS, deal.II and MOOSE ecosystems---provides excellent general-purpose FEM, but none speaks FieldML natively, and none encodes the evaluator-graph abstraction that makes FieldML valuable as an \emph{interchange} format.

The experimental setting for which \texttt{pyfieldml} is designed is the day-to-day biomechanics modelling workflow: load a geometry from a bundled or Physiome-Project dataset, inspect its evaluator graph, evaluate fields over elements, pass the mesh to a solver (such as scikit-fem or a FEniCS front-end), visualise the result, and export to VTK/XDMF for further analysis. The package has already been used internally at Universitat Pompeu Fabra to ingest BodyParts3D \cite{bodyparts3d} skeletal geometries into musculoskeletal pipelines, and it powers the first in-browser FieldML viewer through a JupyterLite \cite{jupyterlite} site that requires no local installation.

\section{Software description}

\subsection{Software architecture}

\texttt{pyfieldml} is organised as a stack of layered subpackages (Fig.~\ref{fig:arch}). The lowest layer, \texttt{dom}, is a thin wrapper over \texttt{lxml} \cite{lxml} that parses and writes FieldML XML against the bundled XSD schemas for versions 0.3, 0.4 and 0.5, and implements an up-converter that normalises older documents to the 0.5 shape on read. The \texttt{model} layer turns that XML tree into a semantic object graph whose nodes are typed evaluators, following the nomenclature of Christie et al.~\cite{fieldml_christie_2009}. A \texttt{Document} exposes mapping-style accessors---\texttt{doc.evaluators}, \texttt{doc.region.continuous}, \texttt{doc.region.meshes}---that make the evaluator graph directly navigable from Python.

\begin{figure}[!h]
\centering
\includegraphics[width=0.98\textwidth]{figures/fig1_architecture.png}
\caption{Layered architecture of \texttt{pyfieldml}. Control flows top-to-bottom: user-facing applications (CLI, JupyterLite site, tutorial notebooks) enter through the \texttt{datasets} facade, fan out through user-facing services (builders, validation linter, \texttt{viz}, interop bridges), and collapse into the evaluation engine that drives the typed evaluator graph of the semantic model. The model layer sits on top of four interchangeable array backends (inline, external text, HDF5 dense, HDF5 DOK sparse), which in turn drive an XML DOM with transparent 0.3\,$\rightarrow$\,0.4\,$\rightarrow$\,0.5 up-conversion. All of this rests on the scientific-Python foundations (\texttt{numpy}, \texttt{h5py}, \texttt{lxml}). Interop bridges fan out sideways to \texttt{meshio}, \texttt{PyVista}, XDMF3 and \texttt{scikit-fem}.}
\label{fig:arch}
\end{figure}

Array data (parameter values, connectivity tables) is handled by the \texttt{data} layer, which provides four interchangeable backends---inline text, external text file, dense HDF5 \cite{hdf5,h5py}, and sparse HDF5 dictionary-of-keys---all exposed through a common \texttt{ndarray}-like interface with lazy loading and round-trip guarantees for \texttt{float32}, \texttt{float64}, \texttt{int32} and \texttt{int64}. The \texttt{eval} layer implements the evaluation engine: a plugin-based basis registry (12 Lagrange bases covering line/triangle/quadrilateral/tetrahedron/hexahedron/wedge at linear and quadratic order, and three cubic-Hermite bases on line/quad/hex with per-node scale factors), a spatial locate routine that inverts world $\rightarrow$ xi by Newton iteration, and a Jacobian calculator. \texttt{Field.evaluate(element, xi)}, \texttt{Field.sample(points)} and \texttt{Field.jacobian(element, xi)} are vectorised over NumPy \cite{numpy} arrays.

The user-facing layer contains \texttt{builders} (high-level constructors for common mesh/field patterns), a \texttt{validation} linter that flags orphan evaluators and inverted elements, a \texttt{cli} (\texttt{pyfieldml inspect\,|\,validate\,|\,convert\,|\,plot\,|\,lint\,|\,diff\,|\,bench}), and the \texttt{datasets} model zoo. Interop bridges are implemented as optional extras: \texttt{meshio} (two-way plugin), \texttt{PyVista} (\texttt{doc.plot()}), \texttt{scikit-fem} (returning a \texttt{(Mesh, Basis)} tuple ready for PDE assembly), \texttt{XDMF3}, and an OpenSim-adjacent asset exporter.

\subsection{Software functionalities}

The key functionalities of \texttt{pyfieldml} are: (i) read/write of FieldML 0.3/0.4/0.5 with transparent up-conversion, validated against the bundled XSD; (ii) an evaluator-graph DOM whose typed objects mirror the specification; (iii) an evaluation engine spanning 15 basis functions over six topologies with vectorised \texttt{evaluate}, \texttt{sample} and \texttt{jacobian}; (iv) a ten-dataset bundled model zoo (synthetic \texttt{unit\_cube}, \texttt{femur}, \texttt{rectus\_femoris}; Stanford Bunny \cite{stanford_bunny}; six real BodyParts3D \cite{bodyparts3d} skeletal segments: \texttt{femur\_bodyparts3d}, \texttt{tibia\_left}, \texttt{hip\_bone\_left}, \texttt{vertebra\_l3}, \texttt{scapula}, \texttt{skull}) shipped inside the wheel; (v) interop bridges to meshio, PyVista, XDMF3 and scikit-fem; (vi) a CLI and semantic linter; (vii) a SHA256-stamped reproducibility benchmark and a conformance workflow that cross-checks I/O against the C++ reference; (viii) an in-browser JupyterLite demo.

\subsection{Sample code snippet}

A complete load $\rightarrow$ evaluate $\rightarrow$ export round-trip fits in ten lines:

\begin{lstlisting}[style=pystyle]
import pyfieldml as fml

doc = fml.datasets.load("femur_bodyparts3d")   # BodyParts3D mesh
coords = doc.field("coordinates")              # evaluator graph
xyz = coords.evaluate(element=1, xi=(0.33, 0.33))
grid = doc.to_pyvista()                        # PyVista mesh
doc.to_meshio().write("femur.vtu")             # VTU export
fml.validation.lint(doc)                       # catch orphans
\end{lstlisting}

No FieldML-specific boilerplate is required; the document, its evaluator graph and its array data behave like first-class Python objects.

\section{Illustrative examples}

The three illustrative examples in this section are drawn from the eight CI-executed tutorial notebooks that ship with \texttt{pyfieldml}.

\subsection{Example 1: The bundled model zoo}

\texttt{pyfieldml} ships with ten curated datasets inside its wheel (Fig.~\ref{fig:zoo}). They range from a 8-node \texttt{unit\_cube} sanity check, through synthetic but anatomically-plausible \texttt{femur} and \texttt{rectus\_femoris} meshes, to the public-domain Stanford Bunny and six real human skeletal segments harvested from BodyParts3D \cite{bodyparts3d}, a CC-BY-SA 2.1 JP anatomical database curated by the Database Center for Life Science (DBCLS) in Japan. A researcher can enumerate, load and visualise the entire zoo in three lines:

\begin{lstlisting}[style=pystyle]
from pyfieldml import datasets
for name in datasets.list():
    datasets.load(name).plot()
\end{lstlisting}

\begin{figure}[!h]
\centering
\includegraphics[width=0.98\textwidth]{figures/fig2_zoo_gallery.png}
\caption{The ten bundled datasets shipped inside the \texttt{pyfieldml} wheel. Top row (synthetic and public-domain): \texttt{unit\_cube}, \texttt{femur}, \texttt{rectus\_femoris}, Stanford Bunny \cite{stanford_bunny}, \texttt{femur\_bodyparts3d}. Bottom row (BodyParts3D \cite{bodyparts3d}, CC-BY-SA 2.1 JP): \texttt{vertebra\_l3}, \texttt{scapula}, \texttt{tibia\_left}, \texttt{hip\_bone\_left}, \texttt{skull}. Node and cell counts shown below each panel.}
\label{fig:zoo}
\end{figure}

The zoo is designed to lower the barrier to reproducible biomechanics experiments: every published pyfieldml workflow can point to a bundled asset rather than to an external download. Each dataset carries a machine-readable \texttt{DatasetInfo} record (SPDX license, citation, origin URL) enabling downstream tooling to surface provenance correctly.

Internally, a \texttt{rectus\_femoris} document exposes an \emph{evaluator graph} in the sense of Christie et al.~\cite{fieldml_christie_2009}: typed evaluators (parameter, external, aggregate, reference, piecewise, argument, constant) composed by name. This graph is navigable from Python (\texttt{doc.evaluators}, \texttt{doc.region.continuous}) and is what makes FieldML valuable as an interchange format. The downstream consumer of that graph is a vector-field workflow (Fig.~\ref{fig:graph}): evaluate the stored fibre direction, integrate streamlines through it, and drive a prescribed contraction---the same evaluate-then-simulate sequence that any Physiome-style biomechanics pipeline performs.

\subsection{Example 2: FEM solve via scikit-fem}

A FieldML document can be handed directly to scikit-fem \cite{scikit_fem} for PDE assembly through the \texttt{pyfieldml.interop.scikit\_fem} bridge, which returns a \texttt{(Mesh, Basis)} pair. The Poisson problem $-\Delta u = 1$ on a unit cube, for instance, reduces to a standard scikit-fem boilerplate once the mesh has been obtained from FieldML (Fig.~\ref{fig:fem}); the figure walks through the full pipeline from refined mesh to scalar solution, gradient magnitude and iso-contours in a single composite. The pattern generalises to any FieldML mesh whose basis is supported by scikit-fem, including the triangular-surface BodyParts3D bones. Crucially, neither the user nor the solver is aware that the mesh began life as a FieldML 0.5 evaluator graph---the bridge is transparent.

\begin{figure}[!h]
\centering
\includegraphics[width=0.98\textwidth]{figures/fig5_fem_solution.png}
\caption{Four-panel FEM workflow on a FieldML mesh. (a) Refined tetrahedral mesh derived from the bundled \texttt{unit\_cube} via the \texttt{pyfieldml $\rightarrow$ scikit-fem} bridge (645 nodes, 2560 tets after three uniform refinements). (b) Poisson solution $-\Delta u = 1$ with zero Dirichlet boundary, rendered on three orthogonal slices through the volume. (c) Gradient magnitude $|\nabla u|$ on a half-clipped volume, showing the boundary-layer gradients. (d) Iso-contours of $u$ in the interior, confirming the bowl-shaped profile expected from the analytic solution.}

\subsection{Example 3: Clinical musculoskeletal assembly}

The most important consumer of \texttt{pyfieldml} is the computational biomechanics community. Fig.~\ref{fig:msk} shows a real-world use case: four BodyParts3D \cite{bodyparts3d} skeletal segments---left hip bone, left femur, left tibia, and lumbar vertebra L3---loaded in a single Python session, composed in a shared coordinate frame, and rendered together. The entire assembly is constructed from bundled datasets, with no external downloads and no C++ build, in roughly ten lines of Python. For MSK researchers this replaces what would otherwise be a multi-tool pipeline involving external STL exporters, manual scaling, and format conversions.

Beyond assembly, the Hermite basis of the evaluation engine enables smooth-deformation studies. Fig.~\ref{fig:hermite} opens with the four cubic-Hermite basis functions on the reference interval (panel~a) and the partition-of-unity property of their value pair (panel~b), then shows a two-node cubic-Hermite line element loaded as a tip-force cantilever (panels~c,d)---the simplest setting in which the $C^1$-continuous Hermite basis is visible. \texttt{pyfieldml} samples the deflection via \texttt{Field.evaluate}; the resulting curve is $C^1$-continuous by construction and matches the theoretical Euler-Bernoulli profile. The same Hermite machinery is what enables smooth heart-wall geometries in Physiome-Project cardiac models \cite{nash_hunter_2001}.

\section{Impact}

\texttt{pyfieldml} has three immediate impacts on the computational-physiology and biomechanics communities.

\textbf{Re-opening the Physiome Project corpus to modern Python.} FieldML is the \emph{lingua franca} of the Physiome Project \cite{hunter_borg_2003,fieldml_christie_2009} and is the native format of a substantial corpus of published models distributed through the Auckland Bioengineering Institute and the OpenCMISS ecosystem \cite{bradley_opencmiss_2011}. That corpus has been locked behind a C++ build for a decade. By providing a pure-Python reader with transparent 0.3 $\rightarrow$ 0.4 $\rightarrow$ 0.5 up-conversion, \texttt{pyfieldml} lets any Python researcher load a FieldML document from the Physiome Model Repository and feed it directly into a modern pipeline (PyVista \cite{pyvista}, scikit-fem \cite{scikit_fem}, FEniCS, OpenSim \cite{delp_opensim_2007,seth_opensim_2018}). The same \texttt{fml.datasets.load(name).to\_meshio()} call that drives this paper's figures suffices.

\textbf{Enabling reproducible musculoskeletal research workflows.} The ten-dataset bundled model zoo is deliberately oriented toward the BodyParts3D \cite{bodyparts3d} skeletal subset most relevant to lower-limb biomechanics. The zoo lets an MSK researcher depend on a single \texttt{pip install pyfieldml} for geometry assets and pair them directly with OpenSim \cite{seth_opensim_2018} muscle-model plugins through the package's OpenSim-adjacent asset exporter (geometry \texttt{.vtu}, muscle attachments CSV, fibre-line polylines). At UPF we have already used this path to prototype fibre-field studies on the bundled \texttt{rectus\_femoris} muscle without needing to distribute mesh assets to collaborators. The new research questions this enables---parameter studies over entire cohorts of BodyParts3D-derived meshes, reproducible benchmarks for MSK pipelines---have been impractical to pursue with the C++ FieldML-API.

\textbf{First in-browser FieldML viewer.} The shipped JupyterLite \cite{jupyterlite} site makes \texttt{pyfieldml} the first FieldML toolchain that can be inspected, evaluated and visualised entirely in a web browser with no local install. This lowers the barrier for teaching (a FieldML tutorial can be run as a URL) and for reviewing: a reviewer of a biomechanics paper can open the authors' JupyterLite sandbox, load the FieldML asset, and re-run the evaluation interactively. For FieldML 0.5, no such capability existed before.

The intended user communities are: the Auckland Bioengineering Institute and the broader Physiome / OpenCMISS community; OpenSim \cite{seth_opensim_2018} and 3D~Slicer \cite{fedorov_3dslicer_2012} practitioners who now consume FieldML assets via meshio; the MSK biomechanics group at Universitat Pompeu Fabra; computational-cardiology groups working with Nash--Hunter-style cardiac mechanics \cite{nash_hunter_2001}; and the broader scientific-Python ecosystem that relies on PyVista \cite{pyvista}, meshio \cite{meshio} and scikit-fem \cite{scikit_fem}. The package is distributed on PyPI with OIDC-backed trusted publishing, and carries a CITATION.cff, Apache 2.0 license, and a continuous-integration matrix covering Python 3.10--3.13 across Linux, macOS and Windows. It has 220+ automated tests and a SHA256-stamped reproducibility benchmark that verifies deterministic output across platforms.

\section{Conclusions}

\texttt{pyfieldml} is a pure-Python reimplementation of the FieldML 0.5 standard that closes the decade-long gap between the conceptual foundation of FieldML \cite{fieldml_christie_2009} and the modern scientific-Python ecosystem. It provides a complete evaluator-graph DOM with 0.3/0.4/0.5 up-conversion, an evaluation engine spanning 15 basis functions on six topologies, a curated ten-dataset musculoskeletal model zoo, interop bridges to meshio, PyVista, XDMF and scikit-fem, a CLI with semantic linter, and the first in-browser FieldML viewer.

The immediate roadmap includes parametric statistical shape models \cite{cootes_ssm_1995} over the BodyParts3D-derived zoo, a full two-way cross-validation workflow against the C++ reference implementation, and native integration with OpenSim's musculoskeletal dynamics pipeline. All code is Apache-2.0 licensed and contributions are welcome at \url{https://github.com/kchemorion/pyfieldml}.
