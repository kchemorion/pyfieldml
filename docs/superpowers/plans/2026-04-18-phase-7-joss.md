# pyfieldml Phase 7 — JOSS Paper Submission Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans (inline execution — this phase is mostly writing, not coding).
>
> **⚠ Forward-draft notice:** Runs in parallel with Phases 5–6. Final submission gated on v1.0.0 tag (§Phase 6 Task 5) and the reproducibility notebook (§Phase 5 Task 10).

**Goal:** Write, submit, and shepherd to acceptance a Journal of Open Source Software (JOSS) paper for pyfieldml. Give Francis and his professor a citable DOI and a peer-reviewed artifact they can cite in grant applications, talks, and downstream papers.

**Architecture:** JOSS papers are short (~2–4 pages) Markdown documents following a prescribed template. Paper lives at `paper/paper.md` with a `paper/paper.bib` BibTeX file. Submission is via the [JOSS webform](https://joss.theoj.org/papers/new) linking to the GitHub repo. Review happens in a public GitHub issue on `openjournals/joss-reviews`. Typical timeline: submission → pre-review triage (~1 week) → reviewer assignment (~2 weeks) → review (~4–6 weeks) → acceptance + DOI.

**Spec reference:** §2.2 Pillar C (JOSS), §7.6 (paper specifics).

**Completion criteria:**
- `paper/paper.md` + `paper/paper.bib` exist and render via JOSS's Docker builder.
- Submission opened on `openjournals/joss-reviews`.
- Reviewer comments all addressed; paper accepted.
- DOI badge added to README and `CITATION.cff` updated with the JOSS DOI.

- [ ] **Step 2: Write the Summary section (≤200 words)**

Target audience: non-specialists. Explain what FieldML is, why it matters, and what pyfieldml adds in two paragraphs.

Skeleton:

```markdown
# Summary

FieldML is a declarative markup language, developed under the Physiome Project, for representing mathematical fields — such as tissue geometry, material heterogeneity, and muscle fiber directions — over discrete finite-element meshes. It underpins computational physiology research within the OpenCMISS ecosystem but has seen little modern tooling outside C/C++ since 2015.

`pyfieldml` is a pure-Python implementation of the FieldML 0.5 specification with a full evaluation engine, interoperability bridges to `meshio`, PyVista, XDMF, and `scikit-fem`, and a curated model zoo focused on musculoskeletal assets. It bridges the Physiome Project's representations into today's scientific-Python ecosystem so that biomechanics researchers can read, construct, evaluate, and visualize FieldML documents with `pip install pyfieldml` rather than with a C++ build.
```

- [ ] **Step 3: Write the Statement of Need section**

Cover:
- The gap: FieldML-API (C++) effectively unmaintained since 2015. No PyPI, no notebooks, no modern CI.
- Why biomechanics users care: muscle fiber directions, heterogeneous bone material, cardiac Hermite meshes.
- What downstream tools benefit: OpenSim workflows, scikit-fem solver integration, meshio/PyVista pipelines.
- Why we did a clean reimplementation instead of binding: ergonomics, legacy burden, modern Python idioms.

Target: ~300 words.

- [ ] **Step 4: Write the Key Features section**

Organize as a short bullet list with section links to docs:
- FieldML 0.5 read + write (with automatic 0.3/0.4 up-conversion).
- Evaluation engine: Lagrange (orders 1–2) + cubic Hermite with scaling on all six standard topologies.
- Programmatic builders: `add_lagrange_mesh`, `add_fiber_field`, `add_material_field`, `add_landmark_set`.
- Interop bridges: meshio, PyVista, XDMF, scikit-fem, OpenSim-adjacent.
- Curated model zoo: `unit_cube`, `femur`, `rectus_femoris`, `myocardium`.
- CLI: `pyfieldml inspect | validate | convert | plot | lint | diff | bench`.
- Conformance-tested against the original C++ FieldML-API reference library.

- [ ] **Step 5: Write the Example usage section**

Include one Python block showing: load, plot, evaluate, export to meshio. Short.

- [ ] **Step 6: Write the Benchmarks / Validation section**

Reference the reproducibility notebook (Phase 5 Task 10). Include ONE figure:

```markdown
![Reproduction of published femoral stiffness predictions (Keller 1994) using pyfieldml + scikit-fem. Solid line: paper values. Markers: our reproduction. Agreement is within XX% across the BMD range. \label{fig:repro}](figures/repro_keller_1994.png)
```

Also mention that every FieldML 0.5 fixture in the reference corpus round-trips byte-semantic-equivalent to the C++ reference library in our conformance CI.

- [ ] **Step 7: Write the Acknowledgments section**

Credit the original FieldML authors (Caton Little, Alan Wu, Richard Christie, Andrew Miller, Auckland Uniservices Ltd), the Physiome Project, and any UPF funding sources.

- [ ] **Step 8: Write the References section — bibliography**

Populate `paper/paper.bib` with:
- Christie et al. "FieldML: concepts and implementation" (2009).
- OpenCMISS citation.
- Keller (1994) or whichever benchmark paper was reproduced.
- scikit-fem, meshio, PyVista, numpy, scipy citations.

- [ ] **Step 9: Commit**

```bash
git add paper/
git commit -m "docs(paper): JOSS paper draft (summary, need, features, example, benchmarks)"
```

## Task 2 — Render locally using the JOSS toolchain

- [ ] **Step 1: Install the JOSS paper-build Docker image**

```bash
docker pull openjournals/inara
```

- [ ] **Step 2: Render the paper to PDF**

```bash
docker run --rm --volume $PWD/paper:/data --user $(id -u):$(id -g) \
  --env JOURNAL=joss openjournals/inara paper.md
```

Output: `paper/paper.pdf`, `paper/paper.jats`.

- [ ] **Step 3: Visual inspection**

- Layout is clean.
- All citations resolve (no "?" markers).
- Figure renders at readable resolution.
- Word count is within JOSS's guidance (ideally 1000–2500 words).

- [ ] **Step 4: Iterate until the PDF looks good.**

- [ ] **Step 5: Commit the rendered PDF (so reviewers can see it at-a-glance)**

```bash
git add paper/paper.pdf
git commit -m "docs(paper): render JOSS PDF for review"
```

## Task 4 — JOSS submission

- [ ] Software is open-source (Apache 2.0 ✅).
- [ ] Software has a DOI (via Zenodo — see Step 2).
- [ ] Paper is at `paper/paper.md` on the default branch.
- [ ] Paper compiles via openjournals/inara.
- [ ] README has clear install + usage.
- [ ] Software has automated tests.
- [ ] Software has a contribution guide.
- [ ] Authors listed correctly.

- [ ] **Step 2: Create a Zenodo release**

Link the GitHub repo to Zenodo at [zenodo.org/account/settings/github](https://zenodo.org/account/settings/github). Toggle the `kchemorion/pyfieldml` switch. On the next GitHub release (the `v1.0.0` tag from Phase 6), Zenodo archives the repo and assigns a DOI. Copy the DOI.

- Branch (`main`).
- Version (`v1.0.0`).
- Zenodo archive DOI.
- Author list.

Submit.
